## Models Deep Dive

### User

The central entity representing an account holder.

| Field | Type | Constraints | Description |
|-------|------|-------------|-------------|
| `id` | Int | PK, auto-increment | Unique identifier |
| `email` | String | Unique | User's email address |
| `password` | String | | Bcrypt hashed password |
| `name` | String | | Display name |
| `householdId` | Int? | FK → Household | Optional household membership |
| `createdAt` | DateTime | Default now() | Account creation |
| `updatedAt` | DateTime | Auto-updated | Last modification |

**Relations:**
- `household` → Household (optional)
- `assignedChores` → Chore[] (tasks assigned to user)
- `paidExpenses` → Expense[] (expenses user paid for)
- `expenseShares` → ExpenseParticipant[] (user's share of expenses)
- `addedNeeds` → HouseholdNeed[] (items user added)
- `purchasedNeeds` → HouseholdNeed[] (items user purchased)
- `paymentsMade` → Payment[] (payments sent)
- `paymentsReceived` → Payment[] (payments received)
- `notifications` → Notification[] (user-targeted notifications)
- `actedNotifications` → Notification[] (notifications where user is actor)

### Household

Represents a shared living group. All data is scoped to a household.

| Field | Type | Constraints | Description |
|-------|------|-------------|-------------|
| `id` | Int | PK, auto-increment | Unique identifier |
| `name` | String | | Household display name |
| `inviteCode` | String? | Unique, optional | 8-character alphanumeric code for joining |
| `createdAt` | DateTime | Default now() | Creation timestamp |
| `updatedAt` | DateTime | Auto-updated | Last modification |

**Relations:**
- `users` → User[] (all household members)
- `chores` → Chore[]
- `expenses` → Expense[]
- `needs` → HouseholdNeed[]
- `payments` → Payment[]
- `notifications` → Notification[]

**Invite Code Generation:**
The invite code is auto-generated as an 8-character alphanumeric string when a household is created. The code can be regenerated by the household (implementation-specific).

### Chore

Task management for household responsibilities.

| Field | Type | Constraints | Description |
|-------|------|-------------|-------------|
| `id` | Int | PK, auto-increment | Unique identifier |
| `title` | String | | Task title (required) |
| `description` | String? | Optional | Detailed description |
| `dueDate` | DateTime? | Default: tomorrow | When task should be completed |
| `isComplete` | Boolean | Default: false | Completion status |
| `householdId` | Int | FK → Household | Required household |
| `assignedToId` | Int? | FK → User | Optional assignee |
| `createdAt` | DateTime | Default now() | Creation timestamp |
| `updatedAt` | DateTime | Auto-updated | Last modification |

**Default Due Date:**
New chores default to due tomorrow (`now() + interval '1 day'`) to encourage timely completion.

### Expense

Core entity for tracking shared costs.

| Field | Type | Constraints | Description |
|-------|------|-------------|-------------|
| `id` | Int | PK, auto-increment | Unique identifier |
| `description` | String | | What was purchased |
| `totalAmount` | Float | | Total expense amount |
| `date` | DateTime? | Default: now() | When expense occurred |
| `paidById` | Int | FK → User | Who paid for it |
| `householdId` | Int | FK → Household | Required household |
| `createdAt` | DateTime | Default now() | Creation timestamp |
| `updatedAt` | DateTime | Auto-updated | Last modification |

**Relations:**
- `paidBy` → User (who fronted the money)
- `household` → Household
- `participants` → ExpenseParticipant[] (split details)

**Validation:**
The sum of all `ExpenseParticipant.shareAmount` should equal `totalAmount`. This is enforced at the application level.

### ExpenseParticipant

Junction table linking expenses to users with their share amounts.

| Field | Type | Constraints | Description |
|-------|------|-------------|-------------|
| `id` | Int | PK, auto-increment | Unique identifier |
| `expenseId` | Int | FK → Expense | Parent expense |
| `userId` | Int | FK → User | Participant |
| `shareAmount` | Float | | Amount this user owes |
| `createdAt` | DateTime | Default now() | Creation timestamp |
| `updatedAt` | DateTime | Auto-updated | Last modification |

**Composite Relation:**
This model creates a many-to-many relationship between Expense and User, with the share amount as the relation attribute.

### Payment

Records debt settlements between household members.

| Field | Type | Constraints | Description |
|-------|------|-------------|-------------|
| `id` | Int | PK, auto-increment | Unique identifier |
| `fromUserId` | Int | FK → User | Payer (debtor) |
| `toUserId` | Int | FK → User | Payee (creditor) |
| `amount` | Int | | Amount paid (in RWF - integer units) |
| `householdId` | Int | FK → Household | Required household |
| `createdAt` | DateTime | Default now() | Payment timestamp |
| `updatedAt` | DateTime | Auto-updated | Last modification |

**Relations:**
- `fromUser` → User (via "PaymentsMade" relation name)
- `toUser` → User (via "PaymentsReceived" relation name)
- `household` → Household

**Note on Amount:**
Amount is stored as an integer representing the smallest currency unit (RWF - Rwandan Francs). This avoids floating-point precision issues.

### HouseholdNeed

Shopping list items for the household.

| Field | Type | Constraints | Description |
|-------|------|-------------|-------------|
| `id` | Int | PK, auto-increment | Unique identifier |
| `name` | String | | Item name (required) |
| `quantity` | String? | Optional | Quantity description |
| `category` | String? | Optional | Category grouping |
| `isPurchased` | Boolean | Default: false | Purchase status |
| `householdId` | Int | FK → Household | Required household |
| `addedById` | Int | FK → User | Who added it |
| `purchasedById` | Int? | FK → User | Who marked purchased |
| `purchasedAt` | DateTime? | | When marked purchased |
| `createdAt` | DateTime | Default now() | Creation timestamp |
| `updatedAt` | DateTime | Auto-updated | Last modification |

**Relations:**
- `household` → Household
- `addedBy` → User (via "AddedNeeds")
- `purchasedBy` → User (via "PurchasedNeeds")

### Notification

Activity feed for household events.

| Field | Type | Constraints | Description |
|-------|------|-------------|-------------|
| `id` | Int | PK, auto-increment | Unique identifier |
| `householdId` | Int | FK → Household | Target household |
| `userId` | Int? | FK → User | Target user (null = all household) |
| `message` | String | | Human-readable message |
| `type` | String? | Optional | Event type identifier |
| `isRead` | Boolean | Default: false | Read status |
| `actorId` | Int? | FK → User | Who triggered the action |
| `entityType` | String? | Optional | Related entity type |
| `entityId` | Int? | Optional | Related entity ID |
| `action` | String? | Optional | Action performed |
| `createdAt` | DateTime | Default now() | Creation timestamp |
| `updatedAt` | DateTime | Auto-updated | Last modification |

**Relations:**
- `household` → Household
- `user` → User (optional - if set, only this user sees it)
- `actor` → User (via "NotificationActor" - who performed the action)

## Indexes and Constraints

### Unique Constraints

| Table | Field | Purpose |
|-------|-------|---------|
| User | `email` | Prevent duplicate accounts |
| Household | `inviteCode` | Prevent code collisions |

### Foreign Key Relationships

All foreign keys have cascading behavior:
- **On Delete**: Restrict (prevent deletion of parent if children exist)
- **On Update**: Cascade (update child FK when parent PK changes)

### Implicit Indexes

Prisma automatically creates indexes on:
- Primary keys
- Unique fields
- Foreign key fields

## Prisma Migrations Guide

### Creating Migrations

```bash
# After modifying schema.prisma
cd backend
npx prisma migrate dev --name descriptive_name
```

### Migration Commands

| Command | Purpose | When to Use |
|---------|---------|-------------|
| `prisma migrate dev` | Create and apply migrations | Development, local changes |
| `prisma migrate deploy` | Apply pending migrations | Production deployment |
| `prisma migrate reset` | Reset database and re-run migrations | Development reset only |
| `prisma migrate status` | Check migration status | Any time |

### Migration Best Practices

1. **Never modify existing migration files** after they've been committed
2. **Always test migrations** on a copy of production data before deploying
3. **Use `--create-only`** to review SQL before applying:
   ```bash
   npx prisma migrate dev --create-only
   ```
4. **Back up production data** before running `migrate deploy`

## Database Queries

### Common Query Patterns

**Get household with all members:**
```typescript
prisma.household.findUnique({
  where: { id: householdId },
  include: { users: true }
});
```

**Get expenses with participants:**
```typescript
prisma.expense.findMany({
  where: { householdId },
  include: {
    paidBy: true,
    participants: { include: { user: true } }
  }
});
```

**Get user's balance (payments made minus received):**
```typescript
// Aggregations using Prisma's groupBy and _sum
prisma.payment.groupBy({
  by: ['fromUserId'],
  where: { fromUserId: userId },
  _sum: { amount: true }
});
```

## Performance Considerations

### N+1 Problem Prevention

Always use `include` or `select` to fetch related data in a single query:

```typescript
// BAD: N+1 queries
const chores = await prisma.chore.findMany();
for (const chore of chores) {
  const user = await prisma.user.findUnique({ where: { id: chore.assignedToId } });
}

// GOOD: Single query
const chores = await prisma.chore.findMany({
  include: { assignedTo: true }
});
```

### Pagination

For large datasets, implement cursor-based pagination:

```typescript
const items = await prisma.chore.findMany({
  take: 20,
  skip: cursor ? 1 : 0,
  cursor: cursor ? { id: cursor } : undefined,
  orderBy: { createdAt: 'desc' }
});
```
